01 / Technical Reading

What exists today

Project type
A local OSINT and macro-intelligence dashboard with a server, recurrent sweep cycle, cached run storage and a purpose-built passive-display frontend.
Framework
No SPA framework is used. The backend is express@5.1.0 on Node ESM; the visible frontend is a handcrafted static HTML dashboard.
Routing
Express serves /, /api/data, /api/health and /events, plus static assets from dashboard/public.
Styling
The dashboard styling is inline inside dashboard/public/jarvis.html, with a dense HUD aesthetic built from custom CSS tokens rather than a separate design system.
Frontend stack
The interface pulls external browser libraries directly from CDNs, including GSAP, D3, TopoJSON, Three.js and Globe.gl.
Data pipeline
The app keeps the Crucix source architecture: multiple OSINT, macro, market, space and social feeds orchestrated through apis/ and synthesized for display.
Live behavior
The server runs a full sweep, saves output to runs/latest.json, computes deltas and pushes browser updates over Server-Sent Events.
Alerts & LLM
Telegram, Discord and optional LLM provider layers are built in via config, but they are operational extensions rather than required for the dashboard to run.

02 / Notes

Observed structure

This is not a generic web app. It is a local command-post surface: more terminal installation than website.

  • The README is explicit that GeoStats is a fork and that the underlying intelligence engine remains Crucix.
  • The visible originality sits mainly in the frontend redesign: locked viewport, wallpaper deployment, new market strip, delta sidebar and modal OSINT treatment.
  • The server layer is still substantial: sweep scheduling, health endpoint, SSE broadcast, cached runs and alert hooks are all already wired.
  • The dashboard is monolithic on the frontend side, with a very large jarvis.html acting as both markup, style and interaction layer.
  • The chosen deployment target is fundamentally local-first: http://localhost:3117 rendered inside Wallpaper Engine.

03 / Surface Inventory

Relevant files

  • server.mjs — Express server, sweep orchestrator, SSE broadcaster and startup runtime.
  • dashboard/public/jarvis.html — full-screen dashboard UI with inline CSS and client logic.
  • dashboard/public/loading.html — boot/loading surface before the first sweep completes.
  • apis/briefing.mjs and apis/sources/* — source ingestion and briefing pipeline.
  • crucix.config.mjs — env-driven runtime configuration for port, refresh interval, alerts and LLM provider.
  • lib/delta/* and lib/llm/* — memory, delta logic and optional provider adapters.
  • Dockerfile and docker-compose.yml — container packaging for local service deployment.

04 / Deployment Reading

Operational shape

  • The project expects Node.js 22+ and exposes port 3117 by default.
  • Docker support is present and minimal: install production deps, copy source, expose the port, and health-check /api/health.
  • docker-compose.yml mounts ./runs into the container, which preserves sweep outputs across restarts.
  • This is not a static-host candidate like GitHub Pages; it needs a live Node process and periodic background execution.
  • The user-facing deployment pattern described by the project is local desktop display, not public cloud hosting.

05 / Weight & Residue

What makes the folder heavy

  • node_modules is the main weight at roughly 18.7 MB.
  • The nested project .git folder adds roughly 2.46 MB.
  • docs contributes around 2.23 MB, mostly screenshots.
  • runs already contains generated output and memory snapshots at roughly 0.76 MB.
  • So the copy is relatively compact; unlike some previous folders, this one is not bloated by large build artifacts.
